Now that infrastructure is ready, let's talk about the actual method to automate software engineering using the BMAD method with OpenClaw. First, let me save you some time and tell you what failed. The naive approach was to install OpenClaw locally and have it run the CloudCode CLI, which would then trigger BMAD agents. So you had me talking to OpenClaw, talking to CloudCode, generating a BMAD prompt, which talked to Cloud again, then passing it back It caused massive token duplication and memory leaks The prompts were too large and it led to context overflow Complete nightmare I ran two experiments with variations of this approach In the best case, I gave OpenClaw the full B-MAD master prompt with meta instruction like every 30 minutes, remind yourself of your long term goal. It actually found an idea on its own, created requirements and deployed something to the VPS. So the potential was clearly there. But after about 10 steps, the bm-ad master node crashed from memory overflow. The agent started improvising, loading its own prompts, abandoning the bm-ad structure, falling back to unstructured coding. The second attempt crashed even earlier. Same root cause every time. The wrapper approach generates too many tokens and kills the context. So I needed a fundamentally different integration. Here is what actually works. Instead of running bhmad through a wrapper, I extracted the individual bhmad role prompts, architect, scrum master, developer, reviewer and saved them as files that OpenClaw can natively load as sub-agents. There is nothing really difficult here, just load the steps and workflow prompt into sub-agent instantiated by OpenClaw. Now if you are like me and have used BMAD in VS Code in the past, you probably used it with Cloud Code. Here, it's different. OpenClaw doesn't need to call Cloud Code to get BMAD behavior. It just needs the right system prompts loaded as sub-agents in its own framework. No wrapper, no extra LLM layer, no token duplication.
